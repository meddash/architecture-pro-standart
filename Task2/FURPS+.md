| Код | Требования | Комментарий |
|-----|------------|-------------|
| F | Функциональные (Functionality) | |
| F | Сайт должен показывать новым клиентам список доступных депозитов с актуальными ставками. | Требование из задания для депозитного MVP. Источник ставок должен быть централизован, так как сейчас они ведутся только в XLS. |
| F | Сайт должен позволять новому клиенту подать заявку на депозит, указав Ф. И. О. и номер телефона. | Минимальный сценарий для внешнего канала продаж. |
| F | После заявки на депозит с сайта менеджер кол-центра должен видеть её в своей системе и иметь возможность связаться с клиентом. | Требует интеграции сайта с системой кол-центра или промежуточным сервисом заявок. |
| F | Менеджер кол-центра должен иметь возможность предложить особые условия по депозиту новому клиенту. | Следует из бизнес-процесса; условия не должны вычисляться вручную вне системы. |
| F | Интернет-банк должен показывать клиенту список доступных депозитов с актуальными ставками и персонализированные ставки. | Архитектурно значимо, так как нужны разные типы ставок: общие и персональные. |
| F | Клиент интернет-банка должен иметь возможность подать заявку на депозит, указав счёт списания и сумму. | Основной цифровой сценарий MVP для существующего клиента. |
| F | Подача заявки на депозит в интернет-банке должна подтверждаться СМС-кодом. | Требует использования существующего SMS-функционала без доработки ядра. |
| F | На этапе MVP заявка на депозит должна поступать менеджеру бэк-офиса, который подтверждает условия депозита в АБС. | Это определяет целевой операционный процесс MVP и интеграцию с АБС. |
| F | После подтверждения ставки и после открытия депозита клиент должен получать СМС-уведомления. | Нужны как минимум два бизнес-события уведомления. |
| F | Должен существовать единый системный источник ставок по депозитам взамен XLS-файлов. | Добавленное требование. Без него нельзя обеспечить актуальные ставки в сайте, интернет-банке и бэк-офисе. |
| F | Сотрудники бэк-офиса депозитов и бэк-офиса кредитов должны работать с данными ставок в рамках разграниченных прав доступа. | Добавленное требование. В текущем процессе есть ограничение безопасности между подразделениями. |
| F | Сайт должен показывать новым клиентам кредитные предложения; полная информация об условиях кредита должна поступать из Кредитного конвейера. | Для кредитного направления система-источник условий уже определена. |
| F | Сайт должен позволять новому клиенту подать заявку на кредит, указав Ф. И. О. и номер телефона. | Базовый сценарий внешнего канала. |
| F | Если новый клиент дополнительно указал паспортные данные, система должна запросить скоринг и выдать предварительное решение по кредиту. | Требуется онлайн-интеграция с системой скоринга для внешнего канала. |
| F | После подачи кредитной заявки на сайте клиент должен получать СМС с приглашением прийти в отделение для идентификации и получения кредита. | Отражает целевой клиентский сценарий для новых клиентов. |
| F | Сотрудник отделения должен иметь возможность продолжить работу с уже существующей кредитной заявкой, созданной на сайте. | Требуется единый идентификатор заявки и сквозной маршрут между каналами. |
| F | Для полностью нового клиента сотрудник отделения должен иметь возможность создать новую кредитную заявку. | Сохраняется текущий офлайн-сценарий. |
| F | Интернет-банк должен показывать клиенту список кредитов с актуальными условиями и предодобренные предложения, если они есть. | Нужна публикация персональных предложений в цифровом канале. |
| F | Клиент интернет-банка должен иметь возможность подать заявку на кредит с указанием счёта зачисления средств и иных необходимых данных. | Основной цифровой сценарий для существующего клиента. |
| F | Подача заявки на кредит в интернет-банке должна подтверждаться СМС-кодом. | Требование безопасности и юридической значимости операции. |
| F | После подачи кредитной заявки должен запускаться повторный скоринг по упрощённой процедуре, если ситуация клиента изменилась. | Требование из задания; влияет на процессинг заявок и критерии вызова скоринга. |
| F | Предодобренные кредитные заявки должны обрабатываться сотрудниками по ускоренному маршруту. | Требует отдельного статуса или приоритета в системе обработки заявок. |
| F | Клиент должен получать СМС об изменении статуса кредитной заявки. | Требует событийной модели статусов и шаблонов уведомлений. |
| U | Удобство использования (Usability) | |
| U | Интерфейсы сайта и интернет-банка должны быть максимально удобными для клиента. | Общее бизнес-требование; ниже конкретизировано архитектурно значимыми пунктами. |
| U | Пользователь должен видеть привычные цвета, брендированные элементы и компоненты в соответствии с корпоративной дизайн-системой. | Требование влияет на выбор UI-компонентов и процесс разработки фронтенда. |
| U | Ключевые пользовательские сценарии подачи заявок на депозит и кредит должны выполняться без визита в отделение, если идентификация уже пройдена в интернет-банке. | Добавленное требование. Иначе цифровой канал не даст ожидаемой ценности для существующих клиентов. |
| U | Статусы заявок и результаты операций должны быть понятны пользователю в цифровом канале без необходимости звонка в кол-центр. | Добавленное требование. Снижает операционную нагрузку на поддержку. |
| U | При ошибках подтверждения СМС или временной недоступности внешних систем пользователь должен получать понятное сообщение и инструкции по дальнейшим действиям. | Добавленное требование. Непрозрачные ошибки ухудшат конверсию MVP. |
| R | Надёжность (Reliability) | |
| R | Все новые сервисы должны работать 24/7. | Прямое требование задания. |
| R | Целевая доступность сервисов должна составлять не менее 99,9%. | Прямое требование задания. |
| R | В случае сбоя в основном ЦОД сервисы должны иметь возможность восстановления в резервном ЦОД. | Требование к отказоустойчивости и DR-сценарию. |
| R | Отказ или недоступность АБС не должен приводить к потере заявок, поданных через сайт или интернет-банк. | Добавленное требование. Следует из перегруженности АБС и запрета на прямую синхронную зависимость цифровых каналов от неё. |
| R | Отказ системы кол-центра, SMS-сервиса или скоринга должен обрабатываться через повторные попытки, очереди и отложенную доставку без потери данных. | Добавленное требование. Иначе процесс MVP будет нестабилен. |
| R | Кредитный конвейер и скоринг должны продолжать обработку роста нагрузки без деградации основного бизнес-процесса. | Следует из их роли и требования горизонтального масштабирования. |
| R | Для заявок должны храниться неизменяемые статусы и история переходов между ними. | Добавленное требование. Нужно для расследований, поддержки и юридически значимых операций. |
| P | Производительность (Performance) | |
| P | Отклик пользовательских операций в цифровых каналах должен измеряться миллисекундами. | Прямое ожидание бизнеса. Требование нужно уточнять на этапе детального проектирования, но оно архитектурно значимо уже сейчас. |
| P | Справочные данные, которые сейчас загружаются более секунды, должны быть оптимизированы. | Возможные решения: кэширование, выделенный read-модель слой, предрасчёт витрин. |
| P | Прямой онлайн-вызов API АБС из интернет-банка в новом процессе должен быть исключён или минимизирован. | Это одновременно ограничение и требование производительности/стабильности из-за перегруженной БД АБС. |
| P | Новые цифровые каналы должны поддерживать равномерное горизонтальное масштабирование и распределение запросов между серверами и приложениями. | Прямое требование задания. |
| P | Запросы должны распределяться между ЦОД с учётом сценариев отказа и балансировки нагрузки. | Расширяет требование про масштабирование на уровне инфраструктуры. |
| P | Предодобренные кредитные заявки должны обрабатываться в течение того же рабочего дня. | Явное бизнес-требование; влияет на архитектуру обхода суточного обмена с АБС. |
| P | Нагрузка на систему кредитного скоринга не должна существенно расти за счёт фоновых предрасчётов и нецелевых вызовов. | Прямое ограничение из задания, важно для проектирования сценариев предодобрения. |
| S | Поддерживаемость (Supportability) | |
| S | При доработках нужно максимально использовать технологии, уже существующие в банке. | Базовый принцип выбора стека. |
| S | Предпочтительными СУБД являются MS SQL и Oracle, либо совместимые с текущими платформами решения. | Архитектурное ограничение и требование к сопровождению. |
| S | При выборе новых технологий желательно опираться на уже существующую экспертизу сотрудников банка. | Снижает риски внедрения и сопровождения. |
| S | Для асинхронного взаимодействия при необходимости следует ориентироваться на Kafka как целевую платформу. | Указано в задании как желательное направление на перспективу. |
| S | Нужно подготовить документацию, достаточную для дальнейшего расширения системы. | Прямое требование задания. |
| S | Архитектура должна позволять поэтапный переход интернет-банка в сторону микросервисного подхода в рамках сценариев депозитов и кредитов. | Добавленное требование. Следует из явного указания рассмотреть микросервисную архитектуру, но ограничить её текущей задачей. |
| S | Новые сервисы должны иметь централизованные логи, метрики и трассировку бизнес-операций. | Добавленное требование. Без наблюдаемости сложно сопровождать распределённые интеграции и SLA 24/7. |
| S | Интеграции и контракты обмена должны быть документированы и версионируемы. | Добавленное требование. Особенно важно для сайта, интернет-банка, АБС, кол-центра, скоринга и кредитного конвейера. |
| +R | + Ограничения (Restrictions) | |
| +R | Для новых клиентов депозитный и кредитный процессы не могут быть полностью дистанционными без визита в отделение для идентификации. | Прямое регуляторное ограничение из задания. |
| +R | Чувствительные данные при передаче с сайта и из интернет-банка должны быть защищены шифрованием трафика. | Минимум TLS для внешних и внутренних взаимодействий, где передаются персональные данные. |
| +R | Функционал SMS в ядре системы нежелательно дорабатывать силами подрядчика; бизнес-функция должна по возможности реализовываться силами команды банка. | Жёсткое ограничение на вариант реализации подтверждения и уведомлений. |
| +R | Текущая версия платформы интернет-банка несовместима с Kafka. | Ограничение на немедленное использование Kafka в контуре интернет-банка без дополнительного слоя адаптации или модернизации. |
| +R | База данных АБС уже перегружена и АБС может масштабироваться только вертикально. | Ключевое ограничение, влияющее на все интеграционные решения. |
| +R | Ускорить обмен между базами АБС и Кредитного конвейера и делать его чаще одного раза в сутки невозможно. | Жёсткое ограничение для кредитного процесса; требует обходных решений для same-day обработки. |
| +R | Сотрудники депозитного и кредитного бэк-офиса не должны получать несанкционированный доступ к данным друг друга. | Следует из процессов Task1 и требований безопасности. |
| +R | Решение MVP должно учитывать существование одного основного ЦОД и резервного ЦОД без предположения о полноценном active-active режиме во всём ландшафте. | Добавленное требование. Соответствует текущему состоянию инфраструктуры из задания. |

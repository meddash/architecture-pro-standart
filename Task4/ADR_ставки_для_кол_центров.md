### <a name="_qmphm5d6rvi3"></a>**Решение**
Для нового кейса предлагается использовать сервис ставок, уже введённый в архитектуре MVP депозитов, как единую точку управления и публикации ставок. Вокруг него добавляются два канала распространения:

1. Внутренний канал для системы банка:
система внутреннего кол-центра получает актуальные ставки из сервиса ставок через внутренний интеграционный API или через локальную read-only витрину.

2. Внешний канал для партнёра:
на основе опубликованных ставок формируется регламентная файловая выгрузка в согласованном формате `CSV/XLSX`. Файл передаётся партнёру через выделенный процесс файлового обмена. Партнёр импортирует файл во внешнюю систему и использует его как источник данных для консультаций.

Логика выбора решения:

1. Нельзя снова строить процесс вокруг XLS как первичного источника, потому что задача MVP как раз в устранении ручного ведения ставок.
2. Для внутреннего кол-центра нужен максимально актуальный доступ к ставкам, поэтому ему подходит онлайн-интеграция с сервисом ставок внутри банковского контура.
3. Для партнёра API недоступен, поэтому файловая выгрузка становится отдельным адаптером поверх того же сервиса ставок, а не отдельной точкой редактирования данных.
4. Чтобы не разъезжались данные, файл для партнёра должен формироваться автоматически из того же набора ставок, который видят интернет-банк и внутренний кол-центр.
5. Передача партнёру должна быть отделена от бизнес-логики хранения ставок: сервис ставок отвечает за данные, а экспортный компонент отвечает за преобразование, упаковку, журналирование и доставку.

Основные компоненты решения:

- Сервис ставок: хранение, публикация и версионирование ставок.
- Рабочее место сотрудников банка: управление ставками с разграничением прав между бэк-офисом депозитов и кредитов.
- Адаптер внутреннего кол-центра: API или read-only интеграция для выдачи актуальных ставок в систему кол-центра банка.
- Экспортный компонент ставок: формирует файл выгрузки для партнёра, присваивает версию, сохраняет журнал отправок и инициирует передачу.
- Файловый шлюз/операционный канал обмена: доставляет файл во внешний контур партнёра.
- Мониторинг и журналирование: фиксирует успешные и неуспешные выгрузки, задержки и повторные отправки.

### <a name="_bjrr7veeh80c"></a>**Альтернативы**
Опишите здесь наиболее важные альтернативные решения.

1. Дать внутреннему кол-центру доступ к XLS-файлам со ставками.
Минусы: сохраняется ручной процесс, нельзя гарантировать актуальность, нет контроля версий и высок риск ошибок при консультациях.

2. Передавать партнёру ставки вручную по email в виде вложений.
Минусы: нет гарантии актуальности, нет журналирования и контроля доставки, процесс плохо масштабируется при частых изменениях ставок.

3. Делать отдельный источник ставок специально для кол-центров.
Минусы: появляется дублирование данных и расхождения между интернет-банком, сайтом, бэк-офисом и кол-центрами.

4. Интегрировать партнёрский кол-центр по API.
Минусы: вариант недоступен по условиям задания и не подходит для текущего внешнего контура партнёра.

**Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

1. Партнёрский кол-центр не получает ставки в реальном времени; актуальность зависит от регламента формирования и доставки файла.
2. Файловый обмен сам по себе менее надёжен и менее прозрачен, чем API-интеграция, поэтому нужен отдельный контроль версий и доставки.
3. Если формат файла будет слабо формализован, интеграция с партнёром станет хрупкой и затратной в сопровождении.
4. При частом изменении ставок возрастает риск того, что внутренний кол-центр и партнёрский кол-центр будут видеть разные версии данных в короткий промежуток времени.
5. Если сервис ставок не будет признан единым владельцем справочника ставок организационно, пользователи могут попытаться вести параллельные локальные таблицы.
6. Появляется зависимость от операционного процесса передачи файла партнёру, даже если бизнес-данные уже готовы в банковском контуре.
